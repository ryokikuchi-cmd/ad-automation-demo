# 広告自動化SaaS コンセプト概要（v0.3 / Meta先行）

作成日: 2026-06-09 / 更新: 2026-06-09 / simcle

## ■一言コンセプト

Meta広告の**データ取得→BI化→クリエイティブ分析→改善提案→CR生成案→入稿→日次運用**を一気通貫で自動化する外販型BtoB SaaS。
御社が内製で確立した「4象限CR分析（cr-analysis.md）」「バナー再現プロンプト（banner-prompt.md）」の型を、広告主が自分で使えるプロダクトに載せ替える。

## ■確定した方向性

| 論点 | 決定 |
|------|------|
| 提供形態 | クライアント(広告主)に外販（マルチテナントBtoB SaaS） |
| 自動化の深さ | 入稿まで自動化（※画像/動画は人がアップ→入稿） |
| 対応媒体 | **Meta先行**（Googleは後続フェーズ） |
| CR生成 | バナースクリプト＋コピー案まで生成（最終画像/動画は人が制作） |
| データ層 | **データベース（PostgreSQL等）**。スプシはデータ層に使わない |
| 商談データ | **クライアントがスプシに手動記入 → SaaSが定期インポート** |
| 命名規則 | **入稿自動化によりSaaS側で命名を統一**（標準化問題は解消） |

---

## ■アーキテクチャ（7レイヤー / ⓪〜⑥）

```
⓪ データ連携層   Meta Marketing APIから日次でKPI＋クリエイティブを自動取得
                 → DBにRawデータとして格納（ROBOMAスプシ経由を廃しAPI直結）
        ↓
① BI化層        Rawをピボット集計（標準メトリクス12列で統一）
                 ＋ 商談スプシをインポートして結合
        ↓
② 分析エンジン   cr-analysisを自動実行。広告セット別4象限＋ビジュアル因果分析
        ↓
③ 提案エンジン   象限別ロジックで改善案を自動生成（横展開/訴求/ビジュアル/停止）
        ↓
④ 生成エンジン   banner-promptを自動実行。修正版バナースクリプト＋コピー案を生成
        ↓
⑤ 実行・報告層   人が画像/動画をツールにアップ → CRとして入稿（命名はSaaSが統一付与）
        ↓
⑥ 日次運用層     予算再配分・日次レポート自動化
```

---

## ■データモデル（既存スプシ「キープサーチ様_詳細レポート」を踏襲）

既存スプシの3層構造をDBスキーマに移植する。

### Rawレイヤー（⓪ API直格納・最小粒度）

| 元シート | カラム | API由来メトリクス |
|--------|--------|------|
| Raw_媒体配置 | アカウント/日付/CP/広告セット/広告/デバイス/媒体/配置/ビジュアル/訴求軸 | 費用/表示/クリック/リード |
| Raw_年齢性別 | アカウント/日付/CP/広告セット/広告/年齢/性別/ビジュアル/訴求軸 | 費用/表示/クリック/リード |

- `ビジュアル` `訴求軸` は広告名から抽出する派生列（→ 命名規則で担保、後述）

### 商談レイヤー（① クライアント記入スプシをインポート）

- クライアントが商談実績をスプシに手動記入
- SaaSが定期（日次）でスプシを読み取り → DBの商談テーブルへ取込
- リード（Meta API）と広告/広告セット単位で結合し、商談・商談率・商談単価を算出
- ※将来的にはCRM連携コネクタも検討（MVPはスプシ取込で十分）

### BIレイヤー（① ピボット集計・標準メトリクス12列）

標準メトリクス: `費用 / Imps / CPM / Clicks / CPC / CTR / 資料請求 / 資料請求率 / 資料請求単価 / 商談 / 商談率 / 商談単価`

集計軸: 全体 / 日別 / 日×広告セット / キャンペーン別 / 広告セット別 / **CR別（4象限の入力）** / プレースメント別 / 性別 / 年齢別 / デバイス別 / 訴求別 / ビジュアル別

---

## ■命名規則の標準化（論点解消の仕組み）

- `ビジュアル` `訴求軸` は広告名（例: `興味関心_人材紹介_01_100名以上の応募実績`）から抽出
- **⑤入稿をSaaSが行う＝命名をSaaSが統一付与**できるため、今後作る広告は命名規則が保証される
- 既存広告／外部作成広告のみ、初回に命名パース or AI推定で補完
- → 運用が回るほど分類精度が自動的に上がる構造

---

## ■分析フレーム（Meta / 4象限）

```
            CTR高              CTR低
CPM安   🟢 勝ちCR/横展開      🟡 訴求が刺さってない
CPM高   🟡 顧客のみ反応       🔴 停止候補
```

- 広告セット別にアクティブCRを判定 → セット内中央値でCPM/CTR閾値を算出
- 画像をAI(Vision)で見てビジュアル因果分析（なぜ効くか）
- 商談データがあればCPA・商談率も加味して総合評価（cr-analysis準拠）

---

## ■④生成エンジンの範囲（banner-prompt連携）

- 入力: ③提案エンジンの改善案（象限別ロジック）
- 出力: **修正版バナースクリプト（再現＋修正プロンプト）＋ コピー案**
- 最終画像/動画は人が制作 → ⑤で人がアップロード → SaaSが命名統一して入稿

---

## ■自動入稿の実行可能範囲（⑤）

| アクション | 自動入稿 |
|-----------|---------|
| 予算再配分（勝ちCRへ集中） | ✅ |
| 負けCRの停止 | ✅ |
| 既存CRの横展開（複製） | ✅ |
| 新規バナー差し替え | ⚠️ 人が画像アップ後に入稿 |

全書き込みは監査ログに記録。承認フロー（提案カード単位で承認/却下/編集）を必須化。

## ■開発フェーズ（Meta先行ロードマップ）

| Phase | 内容 |
|-------|------|
| 0 | 設計確定・Meta API審査申請・DBスキーマ設計・商談スプシ取込仕様 |
| 1 | マルチテナント基盤＋Meta OAuth接続＋⓪Rawデータ取得（DB格納） |
| 2 | ①BI化（ピボット＋商談取込）＋②4象限分析＋③提案（読み取りのみ） |
| 3 | ④生成エンジン（banner-prompt連携：スクリプト＋コピー案） |
| 4 | ⑤承認フロー＋自動入稿（予算/停止/複製/命名統一）＋⑥日次運用 |
| 5 | 課金・レポート・自動承認ルール |
| (後続) | Google対応（検索テキストCR＋アセット系）※別途データモデル拡張 |

## ■重大リスク・要検討

- **Meta API審査**: ads_management 等は審査制。Phase0で着手必須
- **トークン/セキュリティ**: 他社アカウント書き込み権限＝暗号化・最小権限・監査必須
- **商談スプシの運用**: フォーマット逸脱・記入漏れの検知（バリデーション）が必要
- **書き込み事故防止**: 予算上限ガード・ドライラン・承認二重化
- **既存広告の命名補完**: 導入初期のビジュアル/訴求分類のAI推定精度

---

## ■次に詰めるべき箇所（優先順）

1. **DBスキーマ設計**（Raw / 商談 / BI集計 / 広告メタデータ）
2. **商談スプシの取込仕様**（フォーマット・結合キー・バリデーション）
3. **課金モデル**（アカウント数課金／広告費連動 等）
4. **技術スタック確定**（フロント/バックエンド/インフラ）
